## Note sur le perimetre

Le brief mentionnait `kindmother-clients` (pluriel) et `kindmother-client-service`. Ces crates **n'existent pas**. Les crates reels sont :
- `kindmother-client` (singulier) -- Client IPC delegation
- `kindmother-db-adapter` -- Utilitaires communs (helpers purs)
- `kindmother-db-key` -- Derivation de cle AES-256 pour SQLCipher
- `kindmother-service` -- Serveur TCP/JSON isole, seul acces DB

Le crate `kindmother-db-adapter` n'est **pas membre du workspace** Cargo.toml mais est reference par `jayshop` et `jaymanga` via `path = "../kindmother-db-adapter"`.

## Analyse architecture : Separation client/service/adapter

**Verdict : CONFORME avec reserves.**

La separation en 4 crates est coherente et bien structuree :

1. **kindmother-db-adapter** : Helpers purs (timestamps, UUID, conversions, hachage). Zero dep DB. Correct.
2. **kindmother-db-key** : Derivation de cle AES-256 via Argon2id. Zero dep DB. Correct.
3. **kindmother-service** : Serveur isole, seul acces DB. Depend de rusqlite directement. Correct.
4. **kindmother-client** : Client IPC delegation. Zero dep rusqlite. Aucun acces DB direct. Correct.

**Point critique** : La duplication du protocole (D2-L1b-03) est le principal defaut architectural. Un crate `kindmother-protocol` partage entre service et client est la solution standard.

## Analyse architecture : Isolation des couches

**Verdict : CONFORME.**

- Le client n'a aucune dependance vers rusqlite -- pas d'acces DB direct possible.
- Le service est le seul a ouvrir des connexions SQLite.
- L'arbitrage est centralise dans le service (pas dans le client).
- Le protocole IPC est JSON/TCP sur localhost uniquement.
