# P2-L2f-D5-01 -- Audit conformite legale JayKonta

**Auditeur** : George (ISO 19011 / CISA / RGPD)
**Date** : 2026-03-02
**Perimetre** : `crates/jaykonta/` (18 fichiers .rs) + `apps/jaykonta/` (11 fichiers .rs)
**Referentiels** : RGPD, NF525, NF203-CI, ISO 19011
**Severite** : CRITICAL = bloquant legal, MAJOR = ecart significatif, MINOR = amelioration recommandee, INFO = observation

---

## Resume executif

**Score conformite legale : 22 / 100**

JayKonta est un logiciel de comptabilite double-entree (Purse = budget personnel, Account = comptabilite entreprise) avec cycle devis-facture-paiement. L'audit revele des **ecarts critiques** sur les trois referentiels applicables.

**Verdict : NON CONFORME -- NE DOIT PAS ETRE DEPLOYE EN PRODUCTION pour un usage comptable reel sans corrections.**

**Constats cles** :
- NF525 **potentiellement applicable** : le mode `Especes` dans `PaymentMethod` et la mention "encaissements" confirment la vocation d'encaissement. Sans NF525 le risque est de 7 500 EUR d'amende par logiciel.
- NF203-CI : **aucun export FEC**, **aucune cloture d'exercice irreversible**, **aucun audit trail des ecritures comptables** (l'audit_log actuel ne trace que les integrations).
- RGPD : donnees personnelles (noms, emails, adresses, SIRET, telephones) stockees dans `counterparties` **sans base legale documentee**, sans registre des traitements, sans mecanisme d'exercice des droits.
- `INSERT OR REPLACE` permet l'ecrasement silencieux d'ecritures validees, violant le principe d'inalterabilite.

## 1. Determination du champ d'application reglementaire

### 1.1. JayKonta fait-il de l'encaissement ?

**Reponse : OUI, le domaine le prevoit explicitement.**

Preuves :
- `crates/jaykonta/src/domain/account.rs` (ligne 359-360) : `PaymentMethod::Especes` est un moyen de paiement supporte.
- `crates/jaykonta/src/data/schema.sql` (ligne 227) : `CHECK(method IN ('virement', 'cb', 'cheque', 'especes', 'autre'))`.
- `crates/jaykonta/src/backend/service.rs` (ligne 643) : operateur `OP-ACCOUNT-BILLING` dont la responsabilite est "devis, factures, **encaissements**".

**Consequence** : Si JayKonta est utilise pour enregistrer des encaissements d'un assujetti TVA, **NF525 est obligatoire** (Art. 286 bis CGI). L'amende est de 7 500 EUR par logiciel non conforme.

### 1.2. Referentiels applicables

| Referentiel | Applicable ? | Raison |
|-------------|-------------|--------|
| **NF525** | OUI (conditionnellement) | Si utilise pour encaissement TVA. Le code supporte `Especes` + `Cb`. |
| **NF203-CI** | OUI | Logiciel de comptabilite informatisee (devis, factures, grand livre, paiements). |
| **RGPD** | OUI | Donnees personnelles dans `counterparties` (nom, email, adresse, telephone, SIRET). |

## 4. Points positifs identifies

Malgre le nombre de constats critiques, l'architecture de JayKonta presente des fondations solides :

1. **Table `audit`** : l'audit trail existe en structure SQL avec les bons champs (actor_ref, operation, scope, object_ref, result, payload JSON, horodatage). Le probleme est qu'il n'est pas appele systematiquement.

2. **Separation Purse / Account** : la separation fonctionnelle budget personnel / comptabilite entreprise est pertinente et bien modelisee.

3. **Contrats normatifs (CK-SVC, CK-AUD, CK-OP)** : l'intention de gouvernance est presente dans le code (contrats, operateurs, toolkits). La vision est correcte, l'implementation est incomplete.

4. **`unsafe_code = "forbid"`** dans Cargo.toml : conforme aux regles du projet.

5. **Feature `db-encryption`** : le mecanisme de chiffrement de la base existe, meme s'il est optionnel.

6. **`reference_opaque`** dans les paiements : les references de paiement sont explicitement marquees comme opaques/tokenisees, ce qui est un bon reflexe de securisation.

## 7. Recommandation a la direction du projet (Alicia)

JayKonta est actuellement un **logiciel de gestion financiere en phase de prototypage**. Il n'est **pas un logiciel de comptabilite conforme** au sens des normes francaises et europeennes.

**Deux scenarios sont possibles** :

**Scenario A -- Restreindre le perimetre** : Limiter JayKonta au budget personnel (Purse) et au suivi de gestion (devis/factures sans pretention comptable). Retirer la mention "comptabilite" et la methode de paiement `Especes`. Documenter clairement que JayKonta n'est pas un logiciel de caisse ni un logiciel comptable certifie. Cela elimine l'obligation NF525 et reduit significativement les exigences NF203-CI. Le RGPD reste applicable pour les contreparties.

**Scenario B -- Atteindre la conformite** : Implementer les corrections P0/P1/P2 ci-dessus. Cela represente un chantier T5 estimatif de plusieurs semaines. C'est le scenario necessaire si l'objectif est un logiciel comptable utilisable en production pour un professionnel assujetti TVA.

**Avertissement** : En l'etat actuel, la **simple presence du mot "comptabilite"** dans le nom du logiciel et la possibilite d'enregistrer des encaissements en especes cree un **risque juridique reel** en cas de controle fiscal de l'utilisateur.
