## 2. Findings

### F-01 [CRITICAL] -- Recursion infinie dans `run_blocking` (kindmother_client_db.rs:64)

**Fichier** : `src/data/kindmother_client_db.rs` lignes 61-66
**Code** :
```rust
fn run_blocking<F: std::future::Future>(f: F) -> F::Output {
    match tokio::runtime::Handle::try_current() {
        Ok(handle) => tokio::task::block_in_place(|| handle.block_on(f)),
        Err(_) => run_blocking(f),  // <-- recursion infinie si pas de runtime
    }
}
```

**Impact** : Si cette fonction est appelee sans runtime Tokio actif, elle entre en recursion
infinie (stack overflow). Le `Err(_)` signifie qu'il n'y a pas de runtime -- rappeler la
meme fonction ne changera rien.

**Referentiel** : ISO 25010 -- Fiabilite (maturite, tolerance aux pannes). ISTQB -- Boundary
value analysis (pas de runtime = condition limite non geree).

**Recommandation** : Utiliser le runtime global (`get_runtime()`) dans la branche `Err` :
```rust
Err(_) => get_runtime().block_on(f),
```

### F-02 [MAJOR] -- `unwrap_or(0.0)` avale silencieusement les erreurs DB (kindmother_db.rs:687,694)

**Fichier** : `src/data/kindmother_db.rs` lignes 681-694
**Code** :
```rust
let revenus: f64 = conn
    .query_row(/* ... */)
    .unwrap_or(0.0);  // ligne 687
let depenses: f64 = conn
    .query_row(/* ... */)
    .unwrap_or(0.0);  // ligne 694
```

**Impact** : Une erreur SQL (schema corrompu, lock, etc.) est silencieusement traitee comme
un budget a zero. L'utilisateur ne saura jamais que le calcul a echoue. La signature
de la methode retourne `Result<BudgetSummary, DbError>` mais les erreurs sont avalees
avant de pouvoir les propager.

**Referentiel** : ISTQB -- Equivalence partitioning (erreur DB = classe d'entree non testee).

**Recommandation** : Propager l'erreur avec `?` :
```rust
let revenus: f64 = conn.query_row(/* ... */)?;
```

### F-03 [MAJOR] -- `filter_map(|r| r.ok())` avale les erreurs de mapping (kindmother_db.rs:761,782,809)

**Fichier** : `src/data/kindmother_db.rs` lignes 761, 782, 809

**Impact** : Dans la methode `search_all`, les resultats de `query_map` sont filtres avec
`.filter_map(|r| r.ok())` au lieu de `.collect::<Result<Vec<_>, _>>()`. Si une ligne
est corrompue ou un champ manque, elle est silencieusement ecartee. Le reste des methodes
CRUD du meme fichier utilise correctement `.collect::<Result<Vec<_>, _>>()`.

**Referentiel** : ISO 25010 -- Fiabilite (coherence de traitement des erreurs dans un meme module).

**Recommandation** : Utiliser `.collect::<Result<Vec<_>, _>>()?` comme dans les autres methodes,
ou au minimum logger un warning pour les lignes ecartees.

### F-04 [MAJOR] -- SHA-256 brut pour hashage mot de passe (kindmother_db.rs:42-46, kindmother_client_db.rs:68-72)

**Fichier** : `src/data/kindmother_db.rs` lignes 42-46, `src/data/kindmother_client_db.rs` lignes 68-72
**Code** :
```rust
fn hash_password(password: &str) -> String {
    let mut h = Sha256::new();
    h.update(password.as_bytes());
    format!("{:x}", h.finalize())
}
```

**Impact** : SHA-256 sans sel (salt) est vulnerable aux attaques par dictionnaire et rainbow
tables. Ce n'est pas un algorithme de hashage de mot de passe. Le crate miyucloud utilise
correctement Argon2id + HKDF.

**Referentiel** : ISO 25010 -- Securite (confidentialite). OWASP Password Storage Cheat Sheet.

**Recommandation** : Migrer vers `argon2` ou `bcrypt` avec sel aleatoire. Stocker le sel
dans la colonne `password_hash` au format `$argon2id$...`. Prevoir une migration des hashes
existants (re-hashage au prochain login).

### F-08 [MINOR] -- Clones evitables dans auth_sign_in (auth/mod.rs:60-64)

**Fichier** : `src/auth/mod.rs` lignes 60-64
**Code** :
```rust
let user_id = profile.id.clone().unwrap_or_default();
Ok(AuthSession {
    user_id: user_id.clone(),  // clone inutile -- user_id est deja owned
    email: profile.email.clone(),
    access_token: user_id,     // consomme user_id ici
    profile: Some(profile),
})
```

**Impact** : `user_id.clone()` a la ligne 62 est inutile. `user_id` pourrait etre assigne
a `access_token` d'abord (move), puis clone pour `user_id` -- ou simplement clone une
seule fois. Impact negligeable en performance mais indicateur de dette.

De meme dans `auth_sign_up` (lignes 86-88) : `user_id.clone()` suivi de `user_id.clone()`.

**Recommandation** : Restructurer pour minimiser les clones :
```rust
let access_token = user_id.clone();
Ok(AuthSession { user_id, email: profile.email.clone(), access_token, profile: Some(profile) })
```

### F-11 [MINOR] -- `expect()` en production pour creation runtime Tokio (kindmother_client_db.rs:56)

**Fichier** : `src/data/kindmother_client_db.rs` ligne 56
**Code** :
```rust
fn get_runtime() -> &'static tokio::runtime::Runtime {
    RUNTIME.get_or_init(|| {
        tokio::runtime::Builder::new_multi_thread()
            .enable_all()
            .build()
            .expect("Failed to create Tokio runtime")
    })
}
```

**Impact** : Un `expect()` en production. La creation d'un runtime Tokio peut echouer
(ressources systeme insuffisantes). Cependant, sans runtime, aucune operation async n'est
possible, donc un panic est defensable ici -- c'est une pre-condition fatale.

**Referentiel** : Convention Miyukini -- pas de `unwrap()`/`expect()` en production.

**Recommandation** : Acceptable si documente (`// SAFETY: runtime tokio indispensable`).
Alternativement, propager l'erreur a l'appelant via `Result`. Severite MINOR car
l'echec de creation runtime est une condition fatale de toute facon.

### F-12 [MINOR] -- Clippy `allow` excessifs dans Cargo.toml (15 rules)

**Fichier** : `Cargo.toml` lignes 54-71

**Impact** : 15 regles clippy sont desactivees par `allow`. Certaines sont justifiees
(`doc_markdown`, `module_name_repetitions`) mais d'autres masquent potentiellement des
problemes reels :
- `cast_possible_truncation` -- peut cacher des pertes de donnees sur i64 -> usize
- `cast_sign_loss` -- peut cacher des conversions negatives
- `needless_pass_by_value` -- peut cacher des copies inutiles
- `items_after_statements` -- style discutable mais masque la lisibilite

**Referentiel** : ISO 25010 -- Maintenabilite. Convention Miyukini -- clippy pedantic.

**Recommandation** : Revoir les `allow` un par un. Garder `doc_markdown`,
`module_name_repetitions`, `must_use_candidate`, `return_self_not_must_use`.
Supprimer `cast_possible_truncation` et `cast_sign_loss` (remplacer par des
conversions explicites `TryFrom`/`try_into()`). Evaluer les autres cas par cas.

### F-13 [MINOR] -- `candidatures_pending_count` retourne `count as usize` sans verification (kindmother_db.rs:575)

**Fichier** : `src/data/kindmother_db.rs` ligne 575
**Code** :
```rust
Ok(count as usize)
```

**Impact** : Le cast `i64 as usize` est theoriquement sujet a troncation sur plateformes
32-bit (si count > 2^31) et a une perte de signe si count est negatif (impossible ici
grace a COUNT(*) mais le compilateur ne le sait pas). Coherent avec `cast_sign_loss`
et `cast_possible_truncation` qui sont `allow` dans Cargo.toml.

**Recommandation** : Utiliser `usize::try_from(count).unwrap_or(0)` ou laisser tel quel
avec un commentaire expliquant que COUNT(*) ne retourne jamais de negatif. Meme probleme
sur `exposants_count_by_edition` ligne 823.

### F-14 [MINOR] -- `UserType::from_str` masque le trait standard `FromStr`

**Fichier** : `src/data/types.rs` lignes 51-62
**Code** :
```rust
impl UserType {
    #[must_use]
    pub fn from_str(s: &str) -> Self { /* ... */ }
}
```

**Impact** : La methode `from_str` sur un type masque le trait `std::str::FromStr`.
Clippy `should_implement_trait` est normalement declenche par ce pattern. Cela empeche
l'utilisation de `"admin".parse::<UserType>()`.

**Recommandation** : Implementer `impl std::str::FromStr for UserType` a la place,
avec `type Err = ()` (ou un type d'erreur dedie). Garder la methode inherente comme
wrapper `#[must_use]` si necessaire.
