## Analyse detaillee

### 1. Crypto -- SHA-256 pour passwords (S-03a, S-03b)

**Statut S-03 original : PARTIELLEMENT CORRIGE.**

Le crate `kindmother-db-adapter` a ete correctement mis a jour :
- `hash_password()` utilise desormais Argon2id (m=65536, t=3, p=4, 32 bytes output)
- `verify_password()` supporte le format legacy SHA-256 ET le nouveau format Argon2id
- La comparaison utilise `subtle::ConstantTimeEq` pour les deux chemins
- La fonction legacy est marquee `#[deprecated]`
- Le sel Argon2id est genere par `getrandom` (CSPRNG), 16 bytes

**MAIS** les 4 services consommateurs (jayxpose, jayfestival x2, miyukini-central x3) n'ont PAS
ete migres. Ils continuent a :
- Hasher les NOUVEAUX mots de passe en SHA-256 nu (sans sel)
- Comparer avec `==` (non constant-time)
- Comparer dans la clause WHERE SQL (jayxpose)

Cela rend la migration de `kindmother-db-adapter` ineffective : aucun code en production
n'appelle `hash_password()` Argon2id.

### 2. Derivation de cle DB (kindmother-db-key)

**Architecture solide :**
- Argon2id avec parametres OWASP 2024 (m=65536, t=3, p=4)
- Secret d'installation genere par CSPRNG (getrandom), 32 bytes
- Machine ID comme facteur supplementaire (lie au hardware)
- Salt deterministe par nom de DB (necessaire pour reproductibilite)
- `Zeroizing` sur le secret et la cle derivee
- Permissions 0o400 sur le fichier secret (Unix uniquement)

**Defauts identifies :**
- Pas de protection ACL Windows (S-06) : le fichier est lisible par tous
- Material intermediaire non zeroize (S-16)
- Le chiffrement est un feature flag optionnel, pas par defaut (S-09)

### 3. SQL Injection

**Bien protege :**
- Toutes les requetes utilisent `params_from_iter()` (rusqlite parametrees)
- Aucune concatenation de string dans les requetes SQL
- Le code `kindmother-service/database.rs` passe systematiquement les parametres via `?`
- Le protocole transporte les parametres separement du SQL

**Faiblesse de l'arbitrage :**
- La validation SQL par string matching est contournable (S-12)
- Le SQL arbitraire est accepte (S-08) malgre l'arbitrage basique

### 4. Controle d'acces DB

**Architecture d'arbitrage en place :**
- Chaque operateur enregistre avec des permissions par DB (Read/ReadWrite/Admin)
- DDL reserve aux Admin
- Ecriture interdite en Read
- Rate limiting (mais desactive pour tous les operateurs par defaut)

**Faiblesse principale :**
- L'`operator_id` est autodeclare, sans preuve d'identite (S-07)
- Pas d'authentification sur la connexion TCP

### 5. Protection des donnees

- Chiffrement at-rest via SQLCipher avec cle AES-256 derivee (si feature active)
- Pas de donnees personnelles dans les logs d'audit (seul le sql_hash est stocke)
- Permissions fichier restrictives sur Unix
- Connexion localhost uniquement (127.0.0.1)
