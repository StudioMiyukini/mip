## TL;DR

Les 5 services Jay audites presentent un panorama de maturite auth tres heterogene. JayXpose et JayFestival portent des defauts **CRITIQUES** confirmes (SHA-256 nu, comparaison timing-unsafe, absence de politique mot de passe, tokens = user_id). JayManga n'a pas de hashage password propre (auth par COG identity). JayKoa et JayKonta n'ont aucun module auth dedie. L'adaptateur `kindmother-db-adapter` implemente Argon2id + ConstantTimeEq mais **aucun service Jay ne l'utilise pour le hashing**.

## 2. Confirmation S-03 : SHA-256 nu sans sel

### Statut : CONFIRME -- CRITIQUE

**Definition du defaut** : Les fonctions `hash_password()` utilisent `SHA-256` directement sur le mot de passe sans sel (salt), sans iterations, sans key stretching. SHA-256 est un algorithme de hachage rapide, concu pour l'integrite des donnees, PAS pour le stockage de mots de passe. Un attaquant avec acces a la base peut :
- Utiliser des rainbow tables precalculees pour retrouver les mots de passe
- Effectuer des attaques par dictionnaire a ~10 milliards de SHA-256/seconde (GPU moderne)
- Identifier les mots de passe identiques entre utilisateurs (pas de sel)

### Localisation exacte

| Service | Fichier | Ligne | Code |
|---------|---------|-------|------|
| **jayxpose** | `crates/jayxpose/src/auth/mod.rs` | L45-50 | `fn hash_password(password: &str) -> String { Sha256::new(); ... format!("{result:x}") }` |
| **jayfestival** | `crates/jayfestival/src/data/kindmother_db.rs` | L42-46 | `fn hash_password(password: &str) -> String { Sha256::new(); ... format!("{:x}", h.finalize()) }` |
| **jayfestival** | `crates/jayfestival/src/data/kindmother_client_db.rs` | L68-72 | Meme implementation SHA-256 nue |

### Services NON affectes

| Service | Raison |
|---------|--------|
| **jaymanga** | Pas de hashage de mot de passe. Auth par role statique (Admin/Reader/Visitor) et verification de licence COG. SHA-256 utilise uniquement pour l'integrite des fichiers telecharges (`domain/download.rs:61`), ce qui est un usage correct. |
| **jaykoa** | Aucun module auth. Aggregateur de calendriers. |
| **jaykonta** | Aucun module auth. Comptabilite interne. |

### Adaptateur disponible mais non utilise

Le crate `kindmother-db-adapter` (`crates/kindmother-db-adapter/src/lib.rs:185-234`) implemente correctement :
- **Argon2id** avec parametres robustes (65536 Ko memoire, 3 iterations, 4 parallelisme, 32 octets output)
- **Sel aleatoire** de 16 octets (CSPRNG via `OsRng`)
- **Verification legacy** SHA-256 avec migration transparente vers Argon2id
- **Comparaison en temps constant** via `subtle::ConstantTimeEq`

**Probleme** : JayXpose et JayFestival ont leurs propres fonctions `hash_password()` locales qui court-circuitent l'adaptateur. Les hash sont stockes directement comme SHA-256 hex sans passer par l'adaptateur.

### Recommandation

Remplacer toutes les fonctions `hash_password()` locales par des appels a `kindmother-db-adapter::hash_password()` (Argon2id) et `kindmother-db-adapter::verify_password()`. Prevoir une migration des hash existants (verification legacy SHA-256 deja supportee par l'adaptateur).

## 3. Confirmation S-04 : Comparaison timing-unsafe

### Statut : CONFIRME -- ELEVE

**Definition du defaut** : La comparaison de hash avec l'operateur `==` (egalite standard Rust sur `String`) effectue une comparaison byte-a-byte qui s'arrete au premier caractere different. Un attaquant mesurant precisement le temps de reponse peut deduire caractere par caractere le hash attendu (timing attack).

### Localisation exacte

| Service | Fichier | Ligne | Code |
|---------|---------|-------|------|
| **jayfestival** | `data/kindmother_db.rs` | L220 | `if expected == hash_password(password)` |
| **jayfestival** | `data/kindmother_client_db.rs` | L235 | `if expected == hash_password(password)` |

### Cas particulier : JayXpose

JayXpose (`auth/mod.rs:61-64`) pre-hash le mot de passe puis passe le hash a `exposant_by_email_password()` qui effectue une comparaison **en SQL** :
```sql
WHERE contact_email = ?1 AND password_hash = ?2
```
La comparaison est faite par SQLite dans le moteur de requete. Bien que SQLite n'utilise pas non plus une comparaison en temps constant, le timing est domine par le I/O disque et le traitement de la requete, rendant l'exploitation plus difficile mais pas impossible. Ce defaut est classe **MOYEN** pour JayXpose (comparaison SQL) vs **ELEVE** pour JayFestival (comparaison Rust directe).

### Services NON affectes

| Service | Raison |
|---------|--------|
| **jaymanga** | Pas de comparaison de hash password |
| **jaykoa** | Pas d'auth |
| **jaykonta** | Pas d'auth |

### Contexte : miyukini-central egalement affecte

Le service central (`crates/miyukini-central/src/auth/db.rs:264` et `auth/db_client.rs:266`) presente le meme defaut : `if hash == hash_password(password)`. Ce n'est pas dans le perimetre de cet audit mais confirme un pattern systemique.

### Recommandation

Utiliser `subtle::ConstantTimeEq::ct_eq()` pour toute comparaison de hash/secret, ou migrer vers `kindmother-db-adapter::verify_password()` qui l'integre deja.

## 4. Session management et token generation

### Statut : CRITIQUE

**Defaut** : Dans JayXpose et JayFestival, le champ `access_token` de la session est simplement copie depuis le `user_id` (UUID v4 de l'utilisateur).

#### JayXpose (`auth/mod.rs`)

```rust
// Ligne 74-77 (sign_in)
Ok(AuthSession {
    user_id: user_id.clone(),
    access_token: user_id,  // <-- token = user_id
    ...
})

// Ligne 108-112 (sign_up)
Ok(AuthSession {
    user_id: user_id.clone(),
    access_token: user_id,  // <-- token = user_id
    ...
})
```

#### JayFestival (`auth/mod.rs`)

```rust
// Ligne 61-64 (sign_in)
Ok(AuthSession {
    user_id: user_id.clone(),
    access_token: user_id,  // <-- token = user_id
    ...
})

// Ligne 85-89 (sign_up)
Ok(AuthSession {
    user_id: user_id.clone(),
    access_token: user_id.clone(),  // <-- token = user_id
    ...
})
```

### Consequences

1. **Predictibilite** : Si un attaquant connait l'UUID d'un utilisateur (expose dans des logs, URLs, APIs), il possede de facto le token de session.
2. **Pas d'expiration** : Aucun TTL sur le token/session. La session est valide indefiniment.
3. **Pas de rotation** : Le token ne change jamais entre les connexions.
4. **Pas de revocation** : `auth_sign_out()` retourne simplement `Ok(())` sans invalider quoi que ce soit cote serveur.

### Facteur attenuant

Ces services tournent en mode **desktop local** (Dioxus app). La session est en memoire du processus Dioxus, pas dans un cookie HTTP. L'impact est donc attenue car il n'y a pas d'exposition reseau directe. Cependant, si ces services evoluent vers un mode web/API, ce pattern devient immediatement exploitable.

### Recommandation

- Generer un token de session avec CSPRNG (`OsRng` + base64 de 32 octets minimum)
- Ajouter un TTL configurable (ex: 24h par defaut)
- Stocker les sessions actives cote serveur pour permettre la revocation
- Implementer une rotation a chaque re-authentification

## 5. Input validation

### 5.1 Validation email

**Statut** : ABSENT -- MOYEN

Aucune validation de format email n'est effectuee dans les services Jay :
- JayXpose `auth_sign_in()` / `auth_sign_up()` : le parametre `email` est passe directement a la requete SQL sans aucune verification de format.
- JayFestival `profile_create()` / `profile_by_email_password()` : idem.

L'email est utilise comme bind parameter dans les requetes SQL (pas d'injection SQL), mais l'absence de validation permet :
- Creation de comptes avec des emails invalides
- Potentiel d'abus de la base de donnees avec des entrees aberrantes

### 5.2 Injection SQL

**Statut** : NON VULNERABLE

Tous les services utilisent des requetes parametrees (`params![]` pour rusqlite, `vec![]` pour KindMother client). Aucune concatenation de chaine dans les requetes SQL.

**Exception mineure** : `JayFestivalDb::search_all()` (L739) construit `format!("%{}%", query.to_lowercase())` pour le LIKE. La chaine est ensuite passee comme bind parameter, donc pas d'injection SQL. Cependant, les caracteres speciaux LIKE (`%`, `_`) dans la saisie utilisateur ne sont pas echappes, ce qui pourrait donner des resultats de recherche inattendus (defaut fonctionnel, pas securitaire).

### 5.3 Longueur des champs

**Statut** : ABSENT -- FAIBLE

Aucune limite de longueur n'est imposee sur les champs `email`, `password`, `company_name` dans les services Jay. Un attaquant pourrait soumettre des chaines de taille arbitraire. Impact faible car les services sont en mode desktop local.

## 6. Rate limiting

### Statut : ABSENT -- ELEVE (pour JayXpose/JayFestival)

**Constat** : Aucun mecanisme de rate limiting n'existe dans les modules auth des services Jay. Les fonctions `auth_sign_in()` peuvent etre appelees sans limite.

**Impact** : Un attaquant (ou un script automatise) peut tenter un nombre illimite de combinaisons email/password sans etre bloque. Combine avec SHA-256 (rapide) et l'absence de politique de mot de passe, une attaque brute force est facilitee.

**Facteur attenuant** : Mode desktop local. L'attaquant devrait avoir acces physique a la machine ou au processus Dioxus. Impact reclasse de CRITIQUE a ELEVE pour cette raison.

**Note** : Le crate `miyuantispam` dispose d'un module `rate_limit` et le service KindMother a un systeme d'arbitration avec rate limiting, mais ces mecanismes ne sont pas utilises par les couches auth des services Jay.

### Recommandation

Implementer un compteur d'echecs par email avec verrouillage temporaire (ex: 5 tentatives, blocage 15 min). Le crate `miyuantispam::rate_limit_check` pourrait etre integre.

## 7. Password policy

### Statut : ABSENT dans les services Jay -- CRITIQUE

**Constat** : Aucune validation de complexite du mot de passe n'existe dans JayXpose ni JayFestival. L'utilisateur peut creer un compte avec un mot de passe vide (`""`) ou trivial (`"1"`).

| Service | Validation password ? | Politique |
|---------|----------------------|-----------|
| jayxpose | NON | Aucune -- accepte n'importe quelle chaine |
| jayfestival | NON | Aucune -- accepte n'importe quelle chaine |
| jaymanga | N/A | Pas d'auth par password |
| jaykoa | N/A | Pas d'auth |
| jaykonta | N/A | Pas d'auth |
| miyukini-central | OUI | 8 chars min, 1 majuscule, 1 minuscule, 1 chiffre, 1 special (`password.rs`) |

**Contexte** : Le module `miyukini-central/src/auth/password.rs` implemente une politique de mot de passe robuste (`validate_password()`) avec 6 criteres. Ce module n'est pas reutilise par les services Jay, bien qu'il pourrait etre factorise dans un crate commun.

### Recommandation

Extraire `validate_password()` de miyukini-central dans un crate utilitaire partage (ex: `miyukini-auth-common`) et l'integrer dans les `auth_sign_up()` de JayXpose et JayFestival.

## 8. Stockage des credentials

### 8.1 Hashes en base

| Service | Table | Colonne | Format stocke |
|---------|-------|---------|---------------|
| jayxpose | `exposants` | `password_hash` | SHA-256 hex (64 chars) |
| jayfestival | `profiles` | `password_hash` | SHA-256 hex (64 chars) |

Les hash sont stockes en hexadecimal, sans prefixe identifiant l'algorithme. Cela complique la migration progressive vers Argon2id car il n'y a pas de moyen de distinguer un hash legacy d'un hash Argon2id sans convention de prefixe.

**Note** : L'adaptateur `kindmother-db-adapter` utilise le prefixe `$argon2id$` pour les hash Argon2id et detecte les hash legacy par absence de prefixe. Ce mecanisme fonctionnerait si les services migraient vers l'adaptateur.

### 8.2 Sessions en memoire

Les sessions (`AuthSession`) sont stockees en memoire du processus Dioxus. Pas de persistance sur disque, pas de cookie, pas de JWT. Le `access_token` est le `user_id` (voir section 4).

### 8.3 Chiffrement de la base

JayXpose et JayFestival supportent le chiffrement de la base SQLite via le feature flag `db-encryption` et `kindmother-db-key`. Ce mecanisme derive une cle de chiffrement pour le PRAGMA key de SQLCipher. Quand le feature est active, les hash sont proteges at-rest par le chiffrement de la base. Quand il est inactif, les hash SHA-256 sont en clair dans le fichier SQLite.

## 9. Defauts supplementaires detectes

### 9.1 auth_sign_out() no-op

**Gravite** : FAIBLE (contexte desktop)

JayXpose et JayFestival implementent `auth_sign_out()` comme une fonction vide :
```rust
pub fn auth_sign_out() -> AuthResult<()> {
    Ok(())
}
```
Il n'y a aucune invalidation de session, aucun nettoyage de token, aucune journalisation de la deconnexion. Dans un contexte desktop, la session est detruite quand le signal Dioxus est reset. Dans un contexte web/API, ce serait un defaut critique.

### 9.2 Permissions JayFestival : fall-through suspect

**Gravite** : FAIBLE

Dans `jayfestival/src/auth/permissions.rs:48-50`, les roles `Exhibitor`, `Volunteer`, `Visitor` et `Unknown` tombent tous dans le meme match arm et se comportent comme `Manager` (acces si `is_manager_of_edition` est vrai). Le commentaire RLS indique que les Exposants ne devraient avoir acces qu'a leurs propres lignes `editions_exposants`. La granularite semble insuffisante mais la verification finale est deleguee a l'appelant.

### 9.3 JayFestival run_blocking recursion infinie

**Gravite** : CRITIQUE (bug, pas securite)

Dans `jayfestival/src/data/kindmother_client_db.rs:61-66` :
```rust
fn run_blocking<F: std::future::Future>(f: F) -> F::Output {
    match tokio::runtime::Handle::try_current() {
        Ok(handle) => tokio::task::block_in_place(|| handle.block_on(f)),
        Err(_) => run_blocking(f),  // <-- recursion infinie si pas de runtime
    }
}
```
Si aucun runtime Tokio n'est disponible, cette fonction entre en recursion infinie (stack overflow). Ce n'est pas un defaut de securite directement, mais un defaut de fiabilite qui pourrait causer un deni de service involontaire.

## 14. Verdict

**DEFAUTS BLOQUANTS** -- 3 defauts critiques (S-03, S-05, S-07) et 3 defauts eleves (S-04, S-06, S-08) doivent etre corriges avant toute exposition des services Jay sur un reseau (meme local). En mode desktop pur (Dioxus app sans API), le risque est attenue par l'absence d'exposition reseau, mais les defauts restent des violations des bonnes pratiques OWASP A02 et A07.

La correction la plus efficiente est de migrer vers `kindmother-db-adapter` pour le hashing (R-01, R-02) et d'extraire un crate `miyukini-auth-common` pour factoriser la validation et la gestion de session (R-03, R-04, R-09).
