## TL;DR

jaykonta presente **2 defauts CRITICAL**, **10 defauts MAJOR** et **8 defauts MINOR**.
Les points les plus urgents sont : un `panic!` en production dans le `Default` du backend,
une recursion infinie dans `kindmother_client_db`, l'usage systematique de `f64` pour
tous les montants financiers (incompatible NF525/NF203), l'absence totale de tests
unitaires, et l'usage de `INSERT OR REPLACE` qui permet la modification silencieuse
d'ecritures comptables sans audit trail.

### CRITICAL

| C01 | CRITICAL | `backend/service.rs:96-98` | **`panic!` en production dans `Default::default()`**. Si l'ouverture de la base echoue, le programme crash inconditionnellement : `Self::new_with_db("jaykonta.db").unwrap_or_else(\|e\| { panic!(...) })`. Un `Default` impl ne devrait jamais paniquer. | Supprimer l'impl `Default` ou la remplacer par un pattern `lazy_static` / `OnceLock` avec gestion d'erreur explicite. | C02 | CRITICAL | `data/kindmother_client_db.rs:89-94` | **Recursion infinie dans `run_blocking()`**. ## Analyse conformite NF525/NF203

jaykonta est declare comme logiciel comptable (`"accounting"` dans les permissions du manifest).
Voici l'evaluation par rapport aux 4 exigences NF525 fondamentales :

| Exigence NF525 | Statut | Commentaire |
|----------------|--------|-------------|
| **1. Inalterabilite des ecritures** | NON CONFORME | `INSERT OR REPLACE` permet la modification silencieuse (M03). Pas de chainage cryptographique. Pas de scellement. Les ecritures peuvent etre modifiees ou supprimees sans trace. |
| **2. Securisation (chainage)** | NON CONFORME | Aucun mecanisme de chainage (hash SHA-256 de l'ecriture precedente). La dependance `sha2` est declaree mais jamais utilisee (m05). |
| **3. Conservation** | PARTIEL | Les donnees sont conservees en SQLite, mais la migration destructrice (M09) peut effacer toutes les ecritures. Pas de backup automatique. `ON DELETE CASCADE` sur les FK permet la suppression en cascade. |
| **4. Archivage / Export FEC** | NON CONFORME | Aucun module d'export FEC (M07). Aucune fonction d'archivage cloture d'exercice. |

**Verdict NF525** : Le crate n'est pas pret pour une conformite NF525. Les fondations sont presentes
(audit trail dans la table `audit`, contrats CK-AUD), mais l'architecture doit etre renforcee
sur l'inalterabilite, le chainage et l'export.

## Analyse audit trail

Le systeme d'audit est partiellement implemente :

| Element | Present | Commentaire |
|---------|---------|-------------|
| Table `audit` en schema SQL | Oui | 7 colonnes + `CHECK(result IN ('ok','error','denied'))` |
| `AuditService` avec `log()` | Oui | Couvre movement.create, budget.create, goal.create, quote.create, invoice.emit, payment.record |
| Audit sur pipeline integration | Oui | Chaque flux CK-INT-01/02/03 journalise via `IntegrationPipeline::audit()` |
| Audit sur mutations directes | **Non** | `close_budget`, `update_goal_progress`, `toggle_recurring`, `delete_recurring` ne sont pas audites (M04, M05) |
| Audit sur modifications | **Non** | `INSERT OR REPLACE` ne genere pas d'entree audit si l'enregistrement existait deja (M03) |
| Protection integrite audit | **Non** | La table `audit` est modifiable par `UPDATE`/`DELETE` SQL direct. Pas de trigger de protection. |

## Analyse precision decimale

Le crate utilise **exclusivement `f64`** pour tous les montants financiers, tant dans les types Rust
que dans le schema SQLite (`REAL`).

Exemples de risque concret :
- `0.1 + 0.2 = 0.30000000000000004` en IEEE 754
- Cumul de 1000 ecritures de 0.01 EUR = 9.999999999999998 au lieu de 10.00
- Les comparaisons `>=` pour determiner si une facture est payee (trigger `trg_invoice_paid_insert`) sont sujettes a des erreurs d'arrondi

**Impact** : erreurs de rapprochement, calculs de TVA incorrects au centime, totaux de bilan non equilibres.

## Synthese des risques par ordre de priorite

1. **C01 + C02** : Crashes en production -- correctifs immediats (< 1h chacun)
2. **M02** : Zero tests -- risque de regression a chaque modification
3. **M01** : f64 pour les montants -- dette architecturale bloquante pour NF525
4. **M03 + M04 + M05** : Inalterabilite et audit -- fondamental pour la conformite comptable
5. **M07** : Export FEC -- exigence legale pour tout logiciel comptable francais
6. **M08** : Numerotation sequentielle -- exigence CGI
7. **M09** : Migration destructrice -- risque perte de donnees en production
