## Observations supplementaires

1. **Architecture propre** : la separation crate/app est bien respectee, le crate ne contient que la logique metier et la persistance, l'app gere uniquement l'UI.

2. **Invariants metier documentes** : les invariants de JayKoa (lecture seule, pas de creation d'evenement externe, pas de resolution de conflits) sont documentes en commentaires dans `types.rs` et `lib.rs`.

3. **Recursion infinie (finding #1)** : c'est le defaut le plus grave. La fonction `run_blocking` dans `kindmother_client_db.rs:48-53` appelle `run_blocking(f)` dans la branche `Err(_)` sans creer de runtime, ce qui provoquera un stack overflow si aucun runtime tokio n'est disponible. En pratique, le risque est faible car l'app Dioxus fournit toujours un runtime, mais le code est objectivement defectueux.

4. **Module orphelin kindmother_db.rs** : ce fichier contient l'ancienne implementation SQLite directe. Il n'est plus reference dans `data/mod.rs` mais il compile car il contient un `#[cfg(test)]` qui est exclu. Cependant, il gonfle le crate sans raison. Il devrait etre sous feature flag ou supprime.

5. **`unwrap_or_else` chaine dans main.rs** : le fallback vers `"."` pour le repertoire de donnees est dangereux car il depend du current working directory du processus, qui varie selon que l'app est lancee depuis un terminal, un raccourci, ou Central.
