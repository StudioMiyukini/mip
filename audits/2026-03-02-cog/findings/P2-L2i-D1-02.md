## TL;DR

14 findings sur 2 composants (crate + app). 0 CRITICAL, 0 unwrap() nu en production, 0 panic!, 0 unsafe. 1 MAJOR (bug logique : branche dead code dans `metric_category` — les metriques "SV-*" sont classees "sessions" au lieu de "services"). 1 MAJOR (zero tests unitaires ou d'integration sur 9 fichiers pub). 3 MINOR (erreurs d'audit silencieusement ignorees, casts `as i64`/`as u32` sans verification de troncature, annotations MSCM absentes 0/16). 5 MINOR supplementaires (allow blanket `missing_docs` sur 3 fichiers, `allow(unused_imports)` en dur, `Duration::days` deprecated dans chrono, Dioxus 0.7 dans app vs 0.6 convention projet, erreurs DB silencieusement `unwrap_or_default` dans l'UI). Le crate est bien structure (errors.rs propre, feature flags corrects, schema SQL complet, `unsafe_code = "forbid"`), mais l'absence totale de tests et le bug de categorisation des metriques sont des problemes serieux pour un service de collecte de donnees.

### crates/miyukiniwatch

| # | Sev | Fichier:Ligne | Description | Recommandation |

| 1 | MAJOR | `collector.rs:58-61` | **Bug logique — branche dead code** : `metric_category()` teste `starts_with('S')` en premier (ligne 58), puis `starts_with("SV")` (ligne 60). La branche "SV" n'est **jamais atteinte** car tout ID commencant par "SV" commence aussi par "S". Les metriques "SV-01", "SV-02", "SV-03" (service ouvert/ferme/duree) sont classees "sessions" au lieu de "services". Consequence : si `prefs.collect_sessions = false` mais `prefs.collect_services = true`, les metriques de services sont rejetees a tort. Inversement, si sessions=true/services=false, les metriques de services passent quand elles ne devraient pas. | **Inverser l'ordre** : tester `"SV"` avant `'S'` dans le match. Ou mieux, utiliser un pattern plus precis : `starts_with("S-")` pour sessions, `starts_with("SV-")` pour services. |
| 2 | MAJOR | `*` (tout le crate) | **Zero tests** : aucun `#[cfg(test)]`, aucun `#[test]`, aucun fichier dans `tests/`. 19 methodes publiques sur `MiyukiniWatchDb`, 3 fonctions publiques sur `MiyukiniWatchCollector`, 4 sur `MiyukiniWatchPresenter`, 2 sur `MiyukiniWatchAggregator`, plus `compute_aggregates()`. La DB accepte `:memory:` (pattern `Connection::open`), donc les tests unitaires sont triviaux a ecrire. | Ajouter au minimum : (a) test `insert_metric` + `count_metrics_for_session`, (b) test `get_prefs` (creation par defaut) + `set_prefs`, (c) test `delete_all`, (d) test `purge_raw_before`, (e) test `compute_aggregates` sur profil vide et avec donnees, (f) test `metric_category` validant que "SV-01" retourne "services". |
| 3 | MINOR | `aggregator.rs:37`, `presenter.rs:28,59` | **Erreurs d'audit silencieusement ignorees** : `let _ = self.db.insert_audit(...)` ignore toute erreur d'ecriture dans la table d'audit. Si le disque est plein ou la DB corrompue, l'utilisateur ne le saura pas et les traces seront perdues. | Logger les erreurs d'audit via `tracing::warn!` au lieu de `let _`. L'audit est une fonctionnalite de transparence — perdre silencieusement des traces est un defaut de fiabilite. |
| 4 | MINOR | `data/db.rs:178-180,306,323,340,418` | **Casts `as u32` / `as i64` sans verification de troncature** : 3 occurrences de `r.get::<_, i32>(n)? as u32` (lignes 178-180) convertissent un i32 SQLite en u32 sans verifier les valeurs negatives. 4 occurrences de `deleted as i64` (lignes 306, 323, 340, 418) convertissent un `usize` (retour de `conn.execute`) en i64, ce qui est techniquement safe sur 64-bit mais non-idiomatique. | Pour les `as u32` : utiliser `u32::try_from(val).unwrap_or(0)` ou `.max(0) as u32`. Pour les `as i64` : utiliser `i64::try_from(deleted).unwrap_or(i64::MAX)` ou accepter le cast documente (usize -> i64 est safe sur 64-bit). |
| 5 | MINOR | `aggregates.rs:109` | **Cast chaine complexe** : `(now - dt.with_timezone(&Local)).num_days().unsigned_abs().min(u32::MAX as u64) as u32`. L'expression est correcte (le `.min()` borne avant le cast), mais la lisibilite est mauvaise et le clippy `cast_possible_truncation` est probablement silenced. | Extraire en variable intermediaire avec un commentaire explicatif. Ex: `let clamped = abs_days.min(u64::from(u32::MAX)); let days = clamped as u32;` |
| 6 | MINOR | `aggregates.rs:5-6` | **`#![allow(missing_docs)]` et `#![allow(unused_imports)]` au niveau fichier** : le `allow(unused_imports)` est suspect — soit les imports sont utilises (retirer le allow), soit ils ne le sont pas (retirer les imports). Le `allow(missing_docs)` est un echappatoire global qui masque le manque de documentation sur les 3 structs data (`SessionSummaryData`, `SessionReturnData`, `ActivityLevelData`). | Retirer `#![allow(unused_imports)]` et verifier. Remplacer `#![allow(missing_docs)]` par des `#[allow(missing_docs)]` cibles ou mieux, documenter les structs. |
| 7 | MINOR | `data/types.rs:6`, `data/db.rs:5` | **`#![allow(missing_docs)]` blanket sur 2 fichiers supplementaires** : `types.rs` et `db.rs` desactivent completement le lint `missing_docs` au niveau fichier alors que le Cargo.toml le configure en `warn`. 19 methodes publiques dans `db.rs` ont des docstrings malgre le allow (coherent), mais les 5 structs dans `types.rs` ne sont pas toutes documentees. | Retirer les `#![allow(missing_docs)]` globaux.
