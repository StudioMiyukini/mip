### Points positifs

- **Zero `unsafe`** : Les 6 Cargo.toml declarent `unsafe_code = "forbid"`, confirme par `#![forbid(unsafe_code)]` dans chaque `lib.rs`.
- **Zero `panic!` explicite** : Aucun appel `panic!()`, `todo!()`, ou `unimplemented!()` dans le code de production.
- **Zero `unwrap()` brut en production** : Toutes les occurrences de `.unwrap()` et `.expect()` sont soit dans des blocs `#[cfg(test)]`, soit dans des `LazyLock` statics avec des patterns invariants (finding #7).
- **MSCM 100%** : 54/54 fichiers annotes avec les 5 annotations obligatoires. Reference de qualite.
- **Couverture de tests solide sur le domaine** : ~200 tests unitaires repartis sur les modules metier. Les crates `miyualicia-automations` (63+ tests), `miyualicia-mqtt` (42+ tests), et `miyualicia` (42+ tests) sont particulierement bien couverts.
- **Types d'erreur explicites** : Chaque crate a son propre enum d'erreur `thiserror` (`DeviceError`, `MqttError`, `HttpError`, `AutomationError`, `AliciaError`, `ApiError`). Pas de `Box<dyn Error>` ni de `anyhow`.
- **Architecture propre** : Separation claire entre toolkits (Strate 6) et services (Strate 7). Les dependances sont unidirectionnelles (API depend de orchestrateur, orchestrateur depend des toolkits).
- **Gestion du retry HTTP** : `HttpDeviceClient` implemente un pattern retry avec backoff sur timeout et erreurs de connexion.
- **JWT bien structure** : Scopes granulaires, verification d'expiration, extraction automatique via axum `FromRequestParts`, separation generation/verification.

### Defauts structurels

- **5 handlers API sans aucun test** (findings #1-5) : 418 lignes de code public expose via REST sans couverture. C'est le deficit le plus important de la suite. Les 8 endpoints (`health`, `state`, `rooms` x3, `devices` x3, `history`) ne sont valides que par le test de construction du routeur (`test_build_router_creates_router`).
- **Authentification Phase 1 ouverte** (finding #6) : Le endpoint `POST /auth/token` accepte tout `client_id`/`secret` non-vide. En l'etat, n'importe quel client obtient un token JWT valide avec des scopes `read` + `write` + `history`.
- **13 `.expect()` dans des statics** (finding #7) : Faible risque car les patterns regex sont invariants, mais non conforme a la regle "pas de `unwrap()`/`expect()` en production".
- **`DeviceState::is_known()` incomplet** (finding #12) : 6 champs sur 11 ne sont pas verifies, ce qui peut produire des faux negatifs.
- **Fallbacks silencieux** (findings #8-10, #13-14) : Plusieurs `unwrap_or_default()` et `unwrap_or()` masquent des erreurs sans les logger. Le pattern recommande est `.inspect_err(|e| tracing::warn!(...)).unwrap_or_default()`.
