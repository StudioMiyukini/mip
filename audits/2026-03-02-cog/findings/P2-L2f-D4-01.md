## Findings

| # | Sev | WCAG/ISO Ref | Fichier:Ligne | Probleme | Impact | Fix |
|---|-----|-------------|---------------|----------|--------|-----|
| 1 | CRITIQUE | WCAG 2.1.1 (A) Keyboard | Tous fichiers | Aucun `tabindex` sur les elements interactifs non-natifs. Les `div` avec `onclick` dans sidebar, les KPI cards, et les `MovementRow` ne sont pas navigables au clavier. | Utilisateurs clavier bloques, impossible de naviguer dans l'interface sans souris. | Ajouter `tabindex: "0"` et `onkeypress` (Enter/Space) sur tout `div` cliquable, ou convertir en `button`. |
| 2 | CRITIQUE | WCAG 2.4.7 (AA) Focus Visible | Tous fichiers | Tous les `button` utilisent `border: none` sans mecanisme de focus visible. Dioxus inline styles ne supportent pas `:focus-visible`. Aucun handler `onfocus`/`onblur` avec style alternatif. | Focus invisible au clavier, utilisateurs perdus dans la navigation. | Implementer un systeme de focus via signal `is_focused` + `onfocus`/`onblur` avec ring visuel conditionnel (ex: `outline: 2px solid #1a9fff`). |
| 3 | CRITIQUE | WCAG 4.1.2 (A) Name/Role/Value | Tous fichiers | Zero attribut ARIA dans toute l'app : pas de `aria-label`, `role`, `aria-pressed`, `aria-selected`, `aria-live`, `role="alert"`. | Technologies assistives ne peuvent pas interpreter les widgets, les onglets, les formulaires. | Ajouter systematiquement `aria-label` sur les boutons icones, `role="tablist"`/`role="tab"` sur les onglets sidebar, `role="alert"` sur les messages d'erreur. |
| 4 | CRITIQUE | WCAG 2.4.1 (A) Bypass Blocks | mod.rs | Pas de mecanisme "skip to content". La sidebar contient 9+ liens de navigation traverses entierement avant le contenu principal. | Navigation clavier fastidieuse (~15 Tab avant le contenu). | Ajouter un lien "skip to main content" en premier element focusable avec ancre vers `role="main"`. |

## Observations specifiques JayKonta

### Points positifs

1. **Formulaire de mouvement** : validation cote client presente avec messages d'erreur textuels (meme si pas ARIA)
2. **Pagination** : les boutons de pagination utilisent correctement l'attribut `disabled` (purse_movements.rs)
3. **Composants natifs** : utilisation de `button` plutot que `div` pour la majorite des actions
4. **Structure RSX** : code propre, zero piege RSX, variables extraites correctement
5. **Separation des vues** : architecture claire avec composants dedies (KpiCard, AmountDisplay, ProgressBar, etc.)

### Points critiques financiers

L'application manipule des donnees financieres. Les problemes d'accessibilite ont un impact amplifie :
- Un utilisateur malvoyant ne distinguant pas les couleurs pourrait confondre revenus et depenses
- L'absence de confirmation sur la suppression de recurrences impacte directement le suivi budgetaire
- Les montants affiches avec `accent_green`/`accent_red` sur fond sombre echouent au test de contraste WCAG AA
