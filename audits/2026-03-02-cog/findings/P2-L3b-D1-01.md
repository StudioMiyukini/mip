## 6. Patterns de concurrence (Arc, Mutex, RwLock)

### 6.1 Architecture observee

| Registre | Type de lock | Clone via | Usage |
|----------|-------------|-----------|-------|
| `AgentRegistry` | `Arc<RwLock<HashMap>>` x2 | Clone (Arc) | CRUD agents + equipes |
| `ContextRegistry` | `Arc<RwLock<HashMap>>` | Clone (Arc) | CRUD bases de contexte |
| `RateLimiter` | `Arc<RwLock<HashMap>>` | Clone (Arc) | Rate-limiting par client |
| `AuditLog` | `Arc<RwLock<Vec>>` | Clone (Arc) | Journal audit actions |
| `SecurityAuditLog` | `Arc<RwLock<Vec>>` | Clone (Arc) | Journal audit securite |
| `NativeBackend` | `Arc<Mutex<NativeState>>` | Clone (Arc) | Moteur inference GGUF |

**Total** : 8 Arc-wrapped locks partages via `ProxyState` (Clone).

### 6.2 Risque de poison

Si un thread panic pendant qu'il detient un lock (ex: via un `unwrap()` a l'interieur de la section critique), le lock est **empoisonne**. Tous les appels suivants a `.lock().unwrap()` ou `.read().unwrap()` / `.write().unwrap()` **paniquent en cascade**, ce qui arrete le serveur entier.

Scenario concret :
1. `SecurityAuditLog::count_by_kind` (ligne 650-654) fait `serde_json::to_string(kind).unwrap_or_default()` et `serde_json::to_string(&e.kind).unwrap_or_default()` a l'interieur d'un `read().unwrap()`. Si le read reussit mais qu'un autre thread a precedemment empoisonne le lock, c'est un panic cascade.
2. `NativeBackend::chat_completion_blocking` detient le Mutex pendant toute l'inference (potentiellement plusieurs secondes). Si le thread panic, toutes les requetes suivantes paniquent.

### 6.3 Recommandation

Remplacer tous les `.unwrap()` sur locks par :

```rust
// Pour RwLock
let guard = self.data.read().unwrap_or_else(|poisoned| poisoned.into_inner());
// Ou mieux, propager une erreur
let guard = self.data.read().map_err(|_| ServiceError::LockPoisoned)?;
```

## 7. Defauts supplementaires identifies

### 7.1 Timing attack sur comparaison de tokens (MAJOR)

| Fichier | Ligne | Code |
|---------|-------|------|
| security.rs | 219 | `if token == expected` |
| proxy.rs | 902 | `v.strip_prefix("Bearer ").unwrap_or(v) == expected_token.as_str()` |

La comparaison `==` sur `&str` court-circuite au premier octet different. Un attaquant peut mesurer le temps de reponse pour deviner le token caractere par caractere.

**Recommandation** : Utiliser `subtle::ConstantTimeEq` ou un accumulateur XOR.
**Note** : Ce defaut est deja identifie dans MEMORY.md (section "Erreurs a ne pas repeter").

### 7.2 Sandbox file_ops : TOCTOU sur canonicalize (MAJOR)

`file_ops.rs:59-69` : La verification de sandbox utilise `canonicalize()` sur le chemin, puis les operations fichier utilisent le `full_path` original (non canonicalise). Entre la verification et l'utilisation, un symlink pourrait etre cree pour pointer hors du sandbox.

De plus, si le fichier n'existe pas encore (cas `write`), `canonicalize()` echoue et la verification est **silencieusement ignoree** (le `if let Ok(canonical)` passe au else sans bloquer).

### 7.3 CORS trop permissif (MINOR)

`proxy.rs:45-48` :
```rust
let cors = CorsLayer::new()
    .allow_origin(Any)
    .allow_methods(Any)
    .allow_headers(Any);
```

Le CORS est completement ouvert (`*`), meme sur les routes protegees. Toute page web peut envoyer des requetes a l'API. Les protections d'origine dans `security.rs` operent au niveau IP, pas au niveau CORS.

### 7.4 Shell exec whitelist contournable (MINOR)

`shell_exec.rs:100-103` : Si `use_shell: true`, la whitelist est **bypassee**. La seule protection restante est la blocklist de patterns (ligne 88-97), qui est facile a contourner (ex: `rm -r /` au lieu de `rm -rf /`).

### 7.5 UUID non-standard pour contextes (MINOR)

`context.rs:297-303` : `uuid_simple()` utilise `SystemTime::now().as_nanos()` comme pseudo-UUID. C'est predictible et pas un UUID v4. Le meme pattern existe dans `inference.rs:649-655`.

### 7.6 `sign_request` panic sur cle invalide (MAJOR)

`security.rs:503` : `HmacSha256::new_from_slice(secret.as_bytes()).expect("HMAC key")` panic si la cle est invalide. Cette fonction est appelee pour les requetes sortantes. Si la config contient une cle corrompue, le serveur crash au premier appel inter-service.
