# P2-L2g-D1-01 — Audit sante code jaymanga crate (part 1)

> **Perimetre** : `crates/jaymanga/` — 20 premiers fichiers .rs (api/, auth/, data/, domain/aggregator.rs, domain/catalog.rs)
> **Auditeur** : Francois (back-end)
> **Date** : 2026-03-02
> **Domaine** : D1 — Sante code

## TL;DR

20 fichiers audites (api/ 7, auth/ 3, data/ 8, domain/ 2). Aucun `unsafe`, aucun `panic!`, aucun `unwrap()` en production. Le crate est globalement bien structure. **Defaut le plus grave** : `kindmother_db.rs` (1454 lignes, 57 fonctions pub) n'a aucun test unitaire. Pas de dossier `tests/` pour les tests d'integration. 19/20 fichiers n'ont pas d'annotations MSCM (seul `lib.rs` est annote). 14 occurrences de `pub fn from_str()` qui shadow le trait standard `FromStr`.

## Findings

| # | Sev | Fichier:Ligne | Description | Recommandation |
|---|-----|---------------|-------------|----------------|
| 1 | **MAJOR** | `data/kindmother_db.rs` (global) | 1454 lignes, 57 fonctions pub CRUD, **zero test unitaire**. C'est le coeur de la persistance du crate. | Ajouter un module `#[cfg(test)]` avec au minimum un test par operation CRUD (create/read/update/delete) pour chaque entite. Utiliser `:memory:` pour les tests en DB. |
| 2 | **MAJOR** | `crates/jaymanga/tests/` | Aucun dossier `tests/` — pas de tests d'integration. Le crate expose 7 modules publics (data, auth, domain, services, api, export, web) sans verification d'integration. | Creer `tests/` avec au minimum un test d'integration par workflow principal (import + publish, achat + licence, aggregation). |
| 3 | **MAJOR** | `auth/license_verify.rs` (global) | 2 fonctions pub (`license_verify_access`, `license_verify_download`) sans aucun test. Ces fonctions controlent l'acces aux contenus payants. | Ajouter des tests avec DB `:memory:` couvrant : acces avec licence active, acces sans licence, acces avec licence revoquee, download autorise/refuse. |
| 4 | **MAJOR** | `auth/permissions.rs` (global) | 3 fonctions pub de controle d'acces (`auth_can_admin`, `auth_can_purchase`, `auth_can_favorite`) sans aucun test. | Ajouter des tests couvrant tous les `UserType` pour chaque fonction de permission. |
| 5 | **MINOR** | `auth/permissions.rs:28` | `pub fn from_str(s: &str) -> Self` shadow le trait standard `std::str::FromStr`. Identique dans 13 autres enums (types.rs, types_payment.rs, types_reader.rs, types_aggregator.rs). Total : 14 occurrences. | Renommer en `pub fn parse(s: &str) -> Self` ou implementer le trait `FromStr` standard. Le shadow rend l'API non-idiomatique et empeche l'utilisation de `"admin".parse::<UserType>()`. |
| 6 | **MINOR** | `api/federation_api.rs:75` | Comparaison lexicographique de timestamps : `e.updated_at.as_str() > since`. Fonctionnel uniquement si les timestamps sont en ISO 8601 strict avec timezone constante (ex: `Z`). Fragile si un timestamp utilise un offset different (`+02:00` vs `Z`). | Utiliser `chrono::DateTime::parse_from_rfc3339()` pour comparer des `DateTime`, comme fait dans `domain/aggregator.rs:125`. |
| 7 | **MINOR** | `lib.rs:1` | `#![allow(missing_docs)]` desactive l'avertissement `missing_docs` pour tout le crate alors que `Cargo.toml` configure `missing_docs = "warn"`. | Retirer le `#![allow(missing_docs)]` et ajouter les docs manquantes progressivement, ou le restreindre aux modules concernes. |
| 8 | **MINOR** | 19 fichiers sur 20 | Absence d'annotations MSCM (`@id`, `@do`, `@role`, `@layer`, `@human`). Seul `lib.rs` possede les 5 annotations. Aucun sous-module n'est annote. | Ajouter les annotations MSCM dans le header de chaque fichier. Prefixe suggere : `jm-data-*`, `jm-auth-*`, `jm-api-*`, `jm-domain-*`. |
| 9 | **MINOR** | `data/kindmother_client_db.rs` (global) | Placeholder stub qui retourne toujours `Err(...)`. Le type `JayMangaDb` de ce fichier a la meme signature que `kindmother_db.rs` mais n'implemente aucune methode CRUD. Les deux features (`legacy-sqlite` / `kindmother-only`) exposent le meme nom `JayMangaDb` avec des capacites completement differentes. | Ajouter un commentaire `// TODO` explicite avec un tracking issue, ou conditionner la compilation pour empecher l'utilisation accidentelle de la feature `kindmother-only`. |
| 10 | **INFO** | `api/federation_api.rs:63` | `federation_is_allowed()` est une fonction triviale (`fn(bool) -> bool`). Zero valeur ajoutee. | Supprimer ou enrichir avec une logique reelle (verification de token, rate limiting, etc.) lors de l'implementation des handlers axum. |
| 11 | **INFO** | `domain/aggregator.rs:32-41` | `AggregatorConfig` dans `domain/aggregator.rs` est un doublon de `data/types_aggregator.rs::AggregatorConfig`. Les deux ont des champs similaires mais avec des types differents (`bool` vs `Option<bool>`, `Vec<String>` vs `Option<String>` JSON). | Unifier en un seul type ou etablir une conversion explicite (`From`/`Into`) entre les deux. |
| 12 | **INFO** | `data/types.rs` (global) | Tous les champs des structs domaine sont `Option<String>` y compris `id`. Cela permet la construction flexible via `Default::default()` mais rend le contrat flou : un `Work` sans `id` est-il valide? | Envisager un pattern builder ou des types distincts (`WorkCreate` sans id, `Work` avec id obligatoire) pour les operations de creation vs lecture. |
