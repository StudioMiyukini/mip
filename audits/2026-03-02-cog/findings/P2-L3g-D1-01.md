## 3. Analyse thematique

### 3.1 Types monetaires (f64 vs i64)

**Situation** : Les crates POS (`miyupossales`, `miyupospayment`, `miyuposanalytics`) utilisent systematiquement `f64` pour les montants financiers. Les crates compta (`miyucptaledger`, `miyuexpense`, `miyutreasury`, `miyucomptareports`) ne manipulent pas encore de montants types (stubs `String`). Le crate `miyukinisales` est le seul a utiliser correctement `i64` centimes.

**Risque** : La representation IEEE 754 double precision (`f64`) ne peut pas representer exactement tous les montants decimaux. `0.1 + 0.2 != 0.3` en flottant. Pour un systeme de caisse ou de comptabilite, cela peut produire des ecarts de centimes sur les totaux, les rapprochements, et les declarations fiscales.

**Recommandation** : Definir un type newtype `Amount(i64)` dans `miyukini-kernel` ou un crate partage (`miyufinance-types`) qui represente les montants en centimes (ou millimes pour multi-devises). Tous les crates financiers doivent l'adopter.

**Occurrences f64** :
- `miyupossales` : 4 occurrences (cash_movement, cash_register, discount, barcode)
- `miyupospayment` : 5 occurrences (cash, check, split x2, terminal)
- `miyuposanalytics` : 4 occurrences (cash x3, shift)
- Total : **13 sites f64** dans les crates POS

### 3.2 Inalterabilite des ecritures et audit trail

**Situation** : Aucun des 11 crates audites n'implemente de journal d'audit (audit trail). Les ecritures comptables, factures, et transitions de statut sont stockees dans des `HashMap` mutables en memoire sans protection contre l'ecrasement, la modification, ou la suppression.

**Exigences** :
- **NF525** (certification logiciel caisse France) : Inalterabilite, securisation, conservation, archivage des donnees.
- **FEC** (Fichier des Ecritures Comptables) : Chaque ecriture doit etre identifiee, datee, non modifiable.
- **Facturation electronique 2026** : Numerotation sequentielle, chainage, integrite.

**Recommandation** : Avant toute mise en production, implementer :
1. Journal append-only avec hash chaine (SHA-256) pour les ecritures et factures
2. Numerotation sequentielle persistante (pas de compteur atomique en memoire)
3. Horodatage + identification utilisateur sur chaque mutation
4. Interdiction de DELETE/UPDATE sur les tables d'ecritures validees

2. **Zero `unwrap()` en production** dans les 11 crates (uniquement en `#[cfg(test)]`).
3. **Zero `panic!`** dans les 11 crates.
4. **Zero `unsafe`** (forbid dans tous les Cargo.toml).
5. **GovernedContext + NoMandate** : Pattern de securite coherent -- toute operation exige un mandat. Applique partout.
6. **MSCM complet** sur 5 crates POS/compta (miyucomptareports, miyuposloyalty, miyupossales, miyupospayment, miyuposanalytics).
