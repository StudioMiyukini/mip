## Calibration notes

### Principes de calibration pour L1a

1. **CRITICAL** = Exploitation directe possible sans authentification, perte de donnees, ou compromission complete. Exemple : secret en dur, SQL injection, CORS *, TLS absent sur endpoint expose.

2. **MAJOR** = Vulneerabilite exploitable avec un acces limite ou dans des conditions specifiques. Exemple : timing attack sur comparaison de secrets, input non valide permettant injection indirecte, logs contenant des donnees sensibles.

3. **MINOR** = Faiblesse de conception qui n'est pas directement exploitable mais augmente la surface d'attaque ou viole un principe de defense en profondeur. Exemple : absence de logging securite, validation manquante sur un parametre interne, nommage trompeur.

4. **INFO** = Observation, point positif, ou recommandation d'amelioration sans risque immediat. Exemple : design reconnu comme simplifie, dependance saine, stub de test accepte.

### Justification des severites

- **Aucun CRITICAL** : Les 3 crates audites sont du code coeur conceptuel/abstrait. Pas de gestion de secrets, pas de crypto, pas d'endpoint reseau, pas de SQL direct, pas de donnees utilisateur en clair. Le kernel est un mecanisme pur, strongfather est un moteur de decision sans effet de bord, kindmother est une abstraction de persistance avec des implementations memoire (tests).

- **Aucun MAJOR** : Les faiblesses identifiees (D3-01 env vars, D3-06 mandate_id, D3-11/D3-12 policy engine simplifie) ne sont pas exploitables dans le contexte actuel car :
  - Le kernel n'est pas expose directement aux utilisateurs
  - Le policy engine est reconnu comme V0.1 (non deploye en production)
  - Le bridge MiyuSQL est un composant interne

- **D3-01 MINOR plutot que MAJOR** : Le chargement de toutes les env vars est un choix de design (kernel = mecanisme, pas politique). Le risque est reel si un produit consommateur expose le trait Config sans filtrage, mais la responsabilite est au produit, pas au kernel.

- **D3-06 MINOR plutot que MAJOR** : Le mandate_id en String est un composant interne. La verification du mandat est la responsabilite de la couche de gouvernance. Toutefois, un type opaque serait plus robuste.

- **D3-10 MINOR plutot que MAJOR** : L'absence de logging est une faiblesse de defense en profondeur, pas une vulnerabilite directe. Les crates sont des abstractions conceptuelles, pas des services exposes.

- **D3-12 et D3-11 INFO plutot que MAJOR** : Le policy engine est explicitement documente comme simplifie (V0.1). Si deploye en production sans implementation reelle, D3-12 serait un CRITICAL (bypass complet du controle d'acces). Le maintien en INFO est conditionne au fait que ce code ne soit PAS en production.
