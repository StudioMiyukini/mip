## Calibration notes

### Severites D2 -- Architecture

**CRITICAL** : Reserve aux violations de strate (dep N vers >N) et violations de LOI d'autonomie. Un seul finding CRITICAL ici : `D2-01` (kindmother depend de miyusql = Core depend Toolkit). C'est la violation architecturale la plus grave car elle casse la pyramide COG. Un Core ne doit JAMAIS dependre d'une strate superieure.

**MAJOR** : Applique quand un pattern architectural obligatoire est absent ou quand l'API publique fuit des details d'implementation. Les 6 MAJOR couvrent :
- Fuite de types miyusql dans l'API publique de kindmother (D2-02)
- Contournement des abstractions kernel (Clock, IdGenerator) (D2-03, D2-04)
- Absence d'AdminCell/GovernedContext dans les 3 crates (D2-05, D2-06, D2-07)

**Seuil de calibration pour Francois (D1 - Sante code)** : Francois devrait utiliser MAJOR pour `unwrap()` en production, patterns non-idiomatiques structurels. MINOR pour clippy warnings, nommage. CRITICAL seulement si un defaut de code casse un invariant systeme (ex: `unsafe` qui bypass le `forbid`).

**Seuil de calibration pour Victor (D3 - Securite)** : Victor devrait utiliser CRITICAL pour toute faille d'injection, bypass d'auth, secret en clair. MAJOR pour absence de validation d'input, timing attacks. MINOR pour headers manquants, logging insuffisant. Le D2-01 (kindmother<->miyusql) a un impact securite indirect : la fuite de types SQL dans le Core facilite l'injection SQL si un adaptateur fait confiance aux types Core.

### Notes supplementaires

1. **miyusql est le "MiyuSQL Toolkit"** (Strate 3 ou superieure selon sa description : "Kit d'outils MiyuSQL"). Il depend de `miyukini-kernel` + `serde`. Ses types (GovernedContext, QueryResult, etc.) sont des types d'execution SQL.

2. **L'AdminCell est absente des 3 crates.** C'est un pattern mentionne dans la convention standard des crates Miyukini (`admin_cell.rs`). Son absence dans les 3 crates fondamentaux est une dette architecturale importante car elle empeche l'audit automatise de la pyramide.

3. **L'API CoreDataAPI est annoncee comme "10 operations" dans la doc mais n'en implemente que 2.** Ce n'est pas bloquant (v0.1) mais le decalage doc/code est un risque de maintenabilite.

4. **ISO 25010 -- Evaluation maintenabilite** :
   - Modularite : Bonne (modules bien separes) -- 7/10
   - Reutilisabilite : Moyenne (types non portables a cause de la fuite miyusql) -- 5/10
   - Analysabilite : Bonne (MSCM 100%, doc exhaustive) -- 9/10
   - Modifiabilite : Moyenne (AdminCell absente, evaluateurs non injectables) -- 5/10
   - Testabilite : Bonne (traits injectables, MemoryStorage pour tests) -- 7/10

5. **ISO 12207 -- Processus** : Les contrats (traits) sont bien definis mais incomplets (CoreDataAPI 2/10). Le processus de gel de signature (`Signature gelée v0.1`) est present dans le kernel, absent dans strongfather et kindmother.
