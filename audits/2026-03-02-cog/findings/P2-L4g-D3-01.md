---

## TL;DR

Origin est un serveur multi-composants (Web HTTP, Relay TLS, Tracker TCP, Admin HTTP) expose sur Internet. Le Relay utilise TLS 1.3 correctement. Le serveur web et le serveur admin sont en HTTP clair sans TLS. Plusieurs comparaisons de secrets utilisent `==` (court-circuit, timing attack). L'endpoint `/api/auth/forum/sync` accepte des profils arbitraires sans authentification. L'endpoint `/api/market/publish` ne verifie pas reellement le token developpeur. L'IP du VPS est hardcodee dans les tests. Le cookie admin n'a pas le flag `Secure`. Pas de header HSTS ni CSP. 5 defauts critiques, 6 eleves, 4 moyens.

## 1. TLS -- Comment est gere le HTTPS ?

### Constat

| Composant | Port | Protocole | TLS |
|-----------|------|-----------|-----|
| **Relay** | 7000 | TCP custom (frames binaires) | TLS 1.3 via `tokio-rustls` + `rustls 0.22` |
| **Web** | 8080 | HTTP/1.1 (parseur maison) | **AUCUN** -- TCP clair |
| **Tracker** | 21000 | TCP custom (frames binaires) | **AUCUN** -- TCP clair |
| **Admin** | 8081 | HTTP/1.1 (parseur maison) | **AUCUN** -- TCP clair (localhost only) |

**Relay** : Configuration TLS correcte. `setup_tls()` dans `relay/server.rs` charge certificats PEM, supporte TLS 1.3 (par defaut) et TLS 1.2 en option. Validation du certificat et de la cle a l'init. Pas de client auth (normal pour ce use case).

**Web** : Le serveur web ecoute en `0.0.0.0:8080` en TCP clair (`TcpListener::bind`). Aucune couche TLS. Si le VPS expose ce port directement, tout le trafic est en clair (credentials forum, tokens market, donnees personnelles).

**Admin** : Ecoute sur `127.0.0.1:8081` par defaut -- bon reflexe de securite, localhost only. Mais le parseur HTTP lit 8192 octets bruts sans validation.

**Tracker** : TCP clair sur `0.0.0.0:21000`. Le protocole contient des `permis_id` qui transitent en clair.

### Defauts

| # | Defaut | Gravite | Fichier |
|---|--------|---------|---------|
| **S-01** | **Web HTTP sans TLS sur Internet** | **CRITIQUE** | `web/server.rs:98-100` |
| **S-02** | **Tracker TCP sans TLS** | ELEVE | `tracker/server.rs:117-124` |

### Recommandations

- **S-01** : Ajouter TLS au serveur web. Soit directement dans le code (rustls comme pour le Relay), soit via un reverse proxy (nginx/caddy avec Let's Encrypt). Le reverse proxy est la solution la plus pragmatique a court terme.
- **S-02** : Le Tracker transmet des `permis_id` -- envisager TLS ou un mecanisme de chiffrement des payloads.

## 2. Auth -- Authentification des utilisateurs

### Constat

**Admin** (`admin/server.rs`) :
- Authentification par email + mot de passe Argon2id. Hash charge depuis `admin.toml`. Correct.
- Session par cookie `admin_session` (token hex 64 chars genere par `rand::random`).
- Rate limiting : 5 tentatives max, blocage 300s. Correct.
- Cookie : `HttpOnly; SameSite=Strict; Path=/admin; Max-Age=<ttl>`. **MAIS pas de flag `Secure`**.

**Forum Auth** (`web/forum_auth.rs`) :
- Validation par email + mot de passe. Argon2id pour les nouveaux, SHA256 legacy avec migration progressive.
- **L'endpoint `/api/auth/forum/sync` est completement ouvert** : n'importe qui peut envoyer un POST pour inserer/modifier un profil (email, password_hash) sans aucune authentification.

**Relay** :
- Authentification par verification de conformite en 3 phases (Core Key, Services, Environment Health). Modele robuste.
- Anti-rejeu par timestamp window (defaut 10s). Correct.

**Market** :
- `verify_developer_token()` ne fait **aucune verification reelle** : il verifie seulement que le token n'est pas vide, puis retourne `Ok(())`. N'importe quel string non-vide est accepte.

### Defauts

| # | Defaut | Gravite | Fichier |
|---|--------|---------|---------|
| **S-03** | **Cookie admin sans flag `Secure`** | ELEVE | `admin/server.rs:250` |
| **S-04** | **`/api/auth/forum/sync` sans authentification** | **CRITIQUE** | `web/forum_auth.rs:233-256` |
| **S-05** | **`verify_developer_token` = no-op** | **CRITIQUE** | `web/market.rs:324-334` |

### Recommandations

- **S-03** : Ajouter `; Secure` au Set-Cookie. Sans cela, le cookie peut transiter en HTTP clair.
- **S-04** : L'endpoint `/api/auth/forum/sync` doit exiger un token d'administration (shared secret, ou signature HMAC). Actuellement, un attaquant peut injecter n'importe quel profil avec n'importe quel hash, puis se connecter.
- **S-05** : Implementer une verification reelle du token developpeur contre un registre persistant. En attendant, restreindre l'endpoint `POST /api/market/publish` aux IPs de confiance ou ajouter un shared secret.

## 3. CORS

### Constat

Aucun header `Access-Control-Allow-Origin` n'est present nulle part dans le code. Le commentaire `admin/server.rs:778` confirme explicitement "pas de CORS" pour l'admin.

**Impact** : Les API JSON (`/api/catalog`, `/api/status`, `/api/market/*`, etc.) ne sont pas accessibles en cross-origin depuis des applications web tierces. C'est une politique restrictive par defaut -- **correct du point de vue securite** (deny by default).

**Attention** : L'absence de CORS ne protege pas contre les requetes simples (GET, POST form-encoded). Les endpoints POST (`/api/auth/forum/validate`, `/api/market/publish`) restent vulnerables au CSRF si le navigateur est utilise.

### Defaut

| # | Defaut | Gravite | Fichier |
|---|--------|---------|---------|
| **S-06** | **Pas de protection CSRF sur les endpoints POST** | MOYEN | `web/server.rs`, `web/forum_auth.rs` |

### Recommandation

- Ajouter un token CSRF ou verifier le header `Origin`/`Referer` pour les endpoints sensibles. Pour les API JSON, exiger `Content-Type: application/json` (pas form-encoded) ce qui bloque les CSRF simples.

## 4. Input Validation

### Constat

**Body size** : Controle par `content_length_from_headers()` avec limite `MAX_BODY = 65_536` octets (`web/server.rs:147`). Les payloads forum sont limites a 4096 octets additionnellement (`forum_auth.rs:202`). Correct.

**Email** : Normalise en lowercase + trim. Pas de validation de format email (mais acceptable pour un login).

**Market search** : La requete `LIKE ?1` avec `%{query}%` est parametree (pas d'injection SQL). **Mais** le `query` vient directement du query string sans URL-decoding ni sanitization de la longueur.

**Path traversal** : Les chemins d'URL sont utilises directement pour le routage (`path_clean`). Le slug pour les sous-domaines est valide par le `SlugRegistry` qui filtre les caracteres (regex `[a-z0-9-]`). Le `content_length_from_headers` parse avec `.min(MAX_BODY)` pour eviter les allocs geantes.

**HTML** : La fonction `html_escape()` (dans `pages.rs:2574`) echappe `&`, `<`, `>`, `"`, `'`. Utilisee de maniere **consistante** dans tout le code HTML genere. Bon point.

### Defaut

| # | Defaut | Gravite | Fichier |
|---|--------|---------|---------|
| **S-07** | **Market search query sans limite de longueur** | FAIBLE | `web/server.rs:234` |

### Recommandation

- Limiter la longueur du parametre `q` (ex: 256 chars max) avant de le passer au SQL LIKE.

## 5. SQL Injection

### Constat

**Toutes les requetes SQL sont parametrees** via `rusqlite::params![]`. Aucune concatenation de string dans les requetes SQL.

Fichiers verifies :
- `web/forum_auth.rs` : `?1`, `?2` pour SELECT, UPDATE, INSERT
- `web/market.rs` : `?1` a `?16` pour INSERT, SELECT, DELETE, UPDATE
- `admin/server.rs` : pas de SQL (sessions en memoire)

**Verdict : CONFORME**. Pas de risque d'injection SQL.

## 6. Rate Limiting

### Constat

Le rate limiter (`relay/rate_limiter.rs`) est bien implemente :

| Composant | Limite | Fenetre | Blocage |
|-----------|--------|---------|---------|
| **Relay** | `tcp_connections_per_ip` (defaut 5000) | 60s | 300s apres 5 echecs |
| **Web** | 60 conn/IP | 60s | 120s blocage |
| **Tracker** | 20 conn/IP | 60s | 300s apres 5 echecs |
| **Admin login** | 5 tentatives | - | 300s blocage |
| **Forum auth** | **AUCUN** | - | - |

**Architecture** :
- Fenetre glissante par IP + par session.
- Cleanup periodique (120s).
- Support Proof of Work (PoW) avec SHA-256 et difficulte configurable (desactive par defaut).
- Blocage progressif apres echecs de registration.

### Defaut

| # | Defaut | Gravite | Fichier |
|---|--------|---------|---------|
| **S-08** | **Pas de rate limiting sur `/api/auth/forum/validate`** | ELEVE | `web/server.rs:202-213` |

### Recommandation

- L'endpoint de validation forum est accessible sans rate limiting specifique au-dela du rate limiter HTTP global (60 conn/IP/min). Un attaquant peut tenter du brute force sur les mots de passe. Ajouter un rate limiter specifique (ex: 10 tentatives/IP/min) avec blocage progressif.

## 7. Headers Securite

### Constat

**Headers presents** (`web/server.rs:390-395`) :

```
X-Content-Type-Options: nosniff
X-Frame-Options: DENY
X-XSS-Protection: 1; mode=block
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: camera=(), microphone=(), geolocation()
```

**Admin** : `X-Content-Type-Options: nosniff` + `X-Frame-Options: DENY` sur toutes les reponses. Correct.

**Headers ABSENTS** :

| Header | Impact |
|--------|--------|
| `Strict-Transport-Security` (HSTS) | Pas de TLS donc pas de HSTS -- coherent mais dangereux |
| `Content-Security-Policy` (CSP) | Pas de CSP. Les pages HTML chargent Google Fonts en externe. Un attaquant qui injecte du contenu pourrait executer du JS arbitraire. |
| `X-XSS-Protection: 1; mode=block` | Present mais **deprecie** en faveur du CSP. Certains navigateurs l'ignorent. |

### Defauts

| # | Defaut | Gravite | Fichier |
|---|--------|---------|---------|
| **S-09** | **Pas de header HSTS** (lie au S-01 : pas de TLS) | ELEVE | `web/server.rs:390-395` |
| **S-10** | **Pas de Content-Security-Policy** | MOYEN | `web/server.rs:390-395` |

### Recommandations

- **S-09** : Apres mise en place du TLS, ajouter `Strict-Transport-Security: max-age=31536000; includeSubDomains`.
- **S-10** : Ajouter un CSP minimal : `Content-Security-Policy: default-src 'self'; style-src 'self' 'unsafe-inline' fonts.googleapis.com; font-src fonts.gstatic.com; script-src 'self'`.

## 9. Tracker P2P -- Securite

### Constat

**Announce** (`tracker/server.rs:296-431`) :
- Verification du `permis_id` contre le `PermisRegistry` partage avec le Relay (R-011). Si le COG n'a pas de permis valide delivre par le Relay, l'annonce est refusee. **Correct**.
- Remplacement automatique des adresses locales (127.0.0.1, localhost) par l'IP reelle du peer. **Correct** (empeche l'annonce d'adresses privees).

**Slug Registry** (`tracker/slug_registry.rs`) :
- Validation des slugs : regex `[a-z0-9-]`, longueur 3-32, liste de mots reserves ("www", "origin", "api", "admin", "relay", "tracker", "ssh", etc.). **Correct**.
- Unicite garantie (HashMap + generation de suffixe en cas de conflit).

**Lobby passwords** :
- Hash Argon2id avec `OsRng` pour le sel. **Correct** (utilise le bon CSPRNG ici).
- Verification par `argon2::Argon2::default().verify_password()`. Correct.

**Protocole Tracker** :
- TCP clair (cf. S-02). Les `permis_id`, `cog_id`, adresses transitent en clair.
- Pas d'authentification du COG au-dela du permis_id. Un attaquant qui intercepte un permis_id peut usurper l'identite d'un COG.

### Defaut

| # | Defaut | Gravite | Fichier |
|---|--------|---------|---------|
| **S-12** | **Permis_id transmis en clair sur le Tracker TCP** | ELEVE | `tracker/server.rs:326` |

### Recommandation

- A moyen terme, envelopper le Tracker dans TLS ou ajouter un mecanisme de signature HMAC des messages avec la session_key derivee au REGISTER.

## 10. Crypto et comparaison de secrets

### Constat

**Algorithmes utilises** :

| Usage | Algorithme | Verdict |
|-------|-----------|---------|
| Hash mot de passe | Argon2id (defaut) | CONFORME |
| Hash legacy forum | SHA256 sans sel (migration progressive) | ACCEPTABLE (temporaire, migration auto) |
| Verification Cores (Phase A/B) | SHA256 | CONFORME pour verification d'integrite |
| Generation session_key (Relay) | `rand::thread_rng()` | **NON CONFORME** (voir ci-dessous) |
| Generation salt (lobby) | `rand::rngs::OsRng` | CONFORME |
| Generation salt (forum) | `rand::thread_rng()` | **NON CONFORME** |
| Generation admin token | `rand::random::<u8>()` | Utilise `thread_rng` sous le capot |
| Generation challenge PoW | `rand::thread_rng()` | **NON CONFORME** |
| Signature Ed25519 | `ed25519-dalek` | Declare mais non utilise (signature trackers = `[0u8; 64]`) |

**Probleme `thread_rng()` pour crypto** :
- `relay/server.rs:749,865` : `rand::RngCore::fill_bytes(&mut rand::thread_rng(), &mut session_key)` -- genere des cles de session avec un PRNG non-cryptographique garanti.
- `web/forum_auth.rs:59` : `SaltString::generate(&mut rand::thread_rng())` -- sel Argon2id avec thread_rng.
- `relay/rate_limiter.rs:299` : Challenge PoW avec thread_rng.
- `rand::thread_rng()` utilise `StdRng` qui est *generalement* adequat, mais la convention securite Miyukini exige `OsRng` ou `getrandom` pour la generation de materiel cryptographique. Le tracker (`server.rs:728`) utilise correctement `OsRng`.

**Comparaison de secrets non constante** :

| Fichier | Ligne | Code | Risque |
|---------|-------|------|--------|
| `web/forum_auth.rs` | 80 | `sha_hex == stored_hash` | **Timing attack** sur SHA256 legacy |
| `relay/verification.rs` | 115 | `provided_hash.as_slice() == expected_hash` | Timing attack sur verification Cores |
| `relay/verification.rs` | 152 | `provided_hash.as_slice() == expected_hash` | Timing attack sur verification Services |
| `relay/rate_limiter.rs` | 292 | `computed.as_slice() == pow_hash` | Timing attack sur verification PoW |

### Defauts

| # | Defaut | Gravite | Fichier |
|---|--------|---------|---------|
| **S-13** | **`thread_rng()` pour generation de session_key et sels crypto** | MOYEN | `relay/server.rs:749,865`, `web/forum_auth.rs:59`, `relay/rate_limiter.rs:299` |
| **S-14** | **Comparaison `==` sur secrets/hashes (timing attack)** | MOYEN | `web/forum_auth.rs:80`, `relay/verification.rs:115,152`, `relay/rate_limiter.rs:292` |
| **S-15** | **Signature tracker = `[0u8; 64]` (placeholder)** | ELEVE | `relay/server.rs:161` |

### Recommandations

- **S-13** : Remplacer `rand::thread_rng()` par `rand::rngs::OsRng` pour toute generation de materiel cryptographique (session keys, sels, challenges).
- **S-14** : Utiliser `subtle::ConstantTimeEq` ou un accumulateur XOR pour comparer les hashes. Ajouter la dependance `subtle`.
- **S-15** : Implementer la signature reelle des adresses tracker avec Ed25519 (la dependance `ed25519-dalek` est deja presente).

## 17. Verdict

**DEFAUTS BLOQUANTS** -- Origin ne doit pas etre expose sur Internet en l'etat.

Les defauts S-04 (`/api/auth/forum/sync` ouvert) et S-05 (`verify_developer_token` no-op) permettent a n'importe quel attaquant d'injecter des profils ou des packages arbitraires. Le defaut S-01 (HTTP sans TLS) expose toutes les communications en clair, y compris les credentials de login forum.

Apres correction des 3 defauts immediats (S-01, S-04, S-05), le score passerait a environ 78/100 (DEFAUTS NON-BLOQUANTS).
