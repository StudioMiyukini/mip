## 1. Verification des defauts connus (F-01, F-02, F-03)

### F-01 -- Download sans verification de session : CORRIGE

**Etat precedent** : Le endpoint `GET /share/{token}/download` ignorait la protection par mot de passe (`has_password: false` dans la validation). Un attaquant connaissant le token pouvait telecharger sans authentification.

**Etat actuel** : Le correctif est en place dans `apps/miyucloud/src/web_surface/share_page.rs` lignes 214-219.

```rust
// F-01 FIX: If link has password, verify session cookie before allowing download
if link.has_password
    && !verify_session_cookie(&headers, &token, &state.config.cog_token)
{
    return (StatusCode::UNAUTHORIZED, "Authentication required").into_response();
}
```

**Mecanisme de session** : HMAC-SHA256 lie au token de partage et au secret serveur (COG token). Cookie `HttpOnly; Secure; SameSite=Strict; Max-Age=3600`. Genere dans `share_auth()` (ligne 152) apres verification Argon2id du mot de passe. Verifie dans `share_download()` via `verify_session_cookie()`.

**Verification** : La comparaison du cookie est en temps constant via `constant_time_eq()` (lignes 72-81) utilisant un accumulateur XOR. Conforme.

**Qualite de l'implementation** :
- Le HMAC utilise SHA-256 avec un prefixe `"miyucloud-session-v1:"` et le secret serveur comme cle.
- Le cookie est scope au path `/share/{token}` : un cookie obtenu pour un lien ne fonctionne pas pour un autre.
- Expiration 1 heure (`Max-Age=3600`).
- `SameSite=Strict` protege contre les attaques CSRF.

**Verdict** : CORRIGE. Implementation correcte et robuste.

### F-02 -- Timing attack sur la comparaison de hash : CORRIGE

**Etat precedent** : `verify_password()` utilisait `.all(|(a,b)| a == b)` qui court-circuite au premier octet different.

**Etat actuel** : Le code dans `crates/miyucloud/src/auth/web_tokens.rs` lignes 103-111 utilise maintenant un accumulateur XOR :

```rust
// Constant-time comparison (XOR accumulator, no short-circuit)
if computed.len() != expected_hash.len() {
    return false;
}
let mut diff = 0u8;
for (a, b) in computed.iter().zip(expected_hash.iter()) {
    diff |= a ^ b;
}
diff == 0
```

**Verification** : L'accumulateur XOR parcourt TOUS les octets sans court-circuit. Le test de longueur en amont est acceptable car les deux tableaux font toujours 32 octets (output Argon2id fixe). La branche early-return sur la longueur ne revele rien de sensible.

**Note** : L'implementation est correcte mais l'utilisation du crate `subtle` (ConstantTimeEq) serait preferable pour une garantie formelle que le compilateur ne reordonnera pas les operations. L'accumulateur XOR en Rust est neanmoins tres difficile a optimiser de maniere a court-circuiter. Risque residuel negligeable.

**Verdict** : CORRIGE.

### F-03 -- Passphrase par defaut hardcodee : CORRIGE

**Etat precedent** : Si `MIYUCLOUD_PASSPHRASE` n'etait pas definie, le serveur demarrait avec `"miyucloud-default-passphrase"`.

**Etat actuel** : Le bootstrap dans `apps/miyucloud/src/main.rs` lignes 242-251 refuse categoriquement de demarrer :

```rust
let passphrase = match config.passphrase.as_deref() {
    Some(p) if !p.is_empty() => p,
    _ => {
        return Err(miyucloud::errors::MiyucloudError::Crypto(
            "MIYUCLOUD_PASSPHRASE environment variable is required. \
             The server refuses to start without an explicit passphrase."
                .into(),
        ));
    }
};
```

**Verification** : Le `config.passphrase` est lu depuis `std::env::var("MIYUCLOUD_PASSPHRASE").ok()` dans `config.rs` ligne 61. Resultat `None` si absent, chaine vide si `MIYUCLOUD_PASSPHRASE=""`. Les deux cas sont rejetes par le guard `Some(p) if !p.is_empty()`.

**Verdict** : CORRIGE. Le serveur refuse de demarrer sans passphrase explicite.

### 2.5 Master key en RAM -- DEFAUT

**Fichier** : `crates/miyucloud/src/crypto/keys.rs`

| Point de controle | Statut | Detail |
|---|---|---|
| Jamais ecrite sur disque | CONFORME | Uniquement dans `KeyManager.master_key` en RAM |
| Debug masque la valeur | CONFORME | `[REDACTED]` dans impl Debug |
| Pas de Serialize | CONFORME | `KeyManager` n'implemente pas `Serialize` |
| Zeroize au drop | **DEFAUT** | `KeyManager` ne fait pas `#[zeroize(drop)]` et n'implemente pas `Drop` pour effacer `master_key` |
| secrecy::Secret wrapper | **DEFAUT** | Le champ `master_key: [u8; 32]` est un tableau brut, pas enveloppe dans `secrecy::Secret<[u8; 32]>` |

**Analyse** : Quand un `KeyManager` est libere (fin de programme ou remplacement), le contenu de `master_key` reste en memoire jusqu'a ce que les pages soient reutilisees par l'allocateur. Un attaquant avec acces a la memoire du processus (dump, cold boot, swap) pourrait extraire la cle.

**Meme probleme dans** :
- `e2e.rs` : `EphemeralKeypair.secret` (`StaticSecret`) -- `x25519-dalek` fait du zeroize en interne via `Zeroize` trait sur `StaticSecret`. CONFORME.
- `streaming.rs` : `StreamingDecryptor.file_key: [u8; 32]` -- pas de zeroize. DEFAUT mineur (cle derivee, pas master key).
- `web_tokens.rs` : `hash` et `computed` tableaux locaux -- effaces a la fin de la fonction stack frame, risque minime.

**Score** : 3/5

## 3. Audit auth et surface web

### 3.1 Comparaison de token COG non constant-time -- DEFAUT

**Fichier** : `apps/miyucloud/src/api/auth.rs` ligne 80

```rust
Some(token) if token == expected => inner.call(req).await,
```

La comparaison du header `X-COG-Token` avec le token attendu utilise l'operateur `==` sur `String`. C'est une comparaison en temps variable (court-circuit au premier octet different).

**Impact** : Un attaquant local sur le reseau localhost (l'API ecoute sur `127.0.0.1`) pourrait theoriquement executer une timing attack pour deviner le token octet par octet.

**Facteurs attenuants** :
- L'API est bindee sur `127.0.0.1` uniquement -- pas accessible depuis l'exterieur.
- Le token est genere aleatoirement (32 bytes, 256 bits d'entropie via OsRng) si non configure.
- Le bruit reseau sur localhost rend les timing attacks difficiles (resolution nanoseconde requise).

**Severite** : MOYEN (risque theorique, attenuant localhost).
**Recommandation** : Utiliser la meme comparaison constant-time que pour les mots de passe.

### 3.2 Absence de headers de securite HTTP -- DEFAUT

**Fichier** : `apps/miyucloud/src/web_surface/mod.rs` et `apps/miyucloud/src/web_surface/share_page.rs`

Aucun header de securite HTTP n'est present sur les reponses de la surface web :

| Header | Present | Recommandation |
|---|---|---|
| `Content-Security-Policy` | NON | `default-src 'self'; script-src 'unsafe-inline'; style-src 'unsafe-inline'` (minimum) |
| `X-Content-Type-Options` | NON | `nosniff` |
| `X-Frame-Options` | NON | `DENY` (empecher le clickjacking) |
| `Strict-Transport-Security` | NON | `max-age=31536000; includeSubDomains` |
| `X-XSS-Protection` | NON | `1; mode=block` (obsolete mais defense in depth) |
| `Referrer-Policy` | NON | `no-referrer` (empecher le leak de tokens dans le referer) |

**Impact** : Sans `X-Frame-Options`, la page de partage peut etre embeddee dans un iframe malveillant (clickjacking). Sans `CSP`, le XSS (F-04 de George) est plus facilement exploitable. Sans `Referrer-Policy`, le token de partage peut fuiter dans les referer headers si la page contient des liens externes.

**Severite** : MOYEN.
**Recommandation** : Ajouter un middleware Tower qui injecte ces headers sur toutes les reponses de la surface web.

### 3.3 XSS dans la page de partage -- PERSISTANT (F-04 de George)

**Fichier** : `apps/miyucloud/src/web_surface/share_page.rs` ligne 318

```rust
<h1>{file_name}</h1>
```

Le nom de fichier est insere directement dans le HTML via `format!()` sans echappement. Ce defaut etait deja signale par George (F-04). Il n'a pas ete corrige.

De plus, `file_size`, `expires_at` et `token` sont egalement inseres sans echappement dans le template (lignes 319, 321-322, 324, 329).

**Verification supplementaire** : Le `Content-Disposition` header dans le download (ligne 257) insere egalement le nom de fichier sans echappement des guillemets :
```rust
let content_disposition = format!("attachment; filename=\"{}\"", file_entry.name);
```
Un nom de fichier contenant `"` pourrait casser le header et potentiellement injecter des headers supplementaires (header injection).

**Severite** : MOYEN.
**Recommandation** : Implementer `html_escape()` pour tous les contenus dynamiques dans les templates HTML. Pour le `Content-Disposition`, utiliser l'encodage RFC 5987 (`filename*=UTF-8''...`).

### S-01 -- ELEVE : Pas de zeroize sur la master key au drop

**Fichiers** : `crates/miyucloud/src/crypto/keys.rs` (KeyManager), `crates/miyucloud/src/crypto/streaming.rs` (StreamingDecryptor)

**Description** : La structure `KeyManager` contient la master key en `[u8; 32]` sans implementer `Drop` pour l'effacer. Quand le `KeyManager` est libere, la cle reste en memoire.

**Impact** : Un attaquant avec acces a la memoire du processus (dump memoire, swap, cold boot attack) pourrait extraire la master key. Risque proportionnel au niveau de menace physique.

**Recommandation** :
```toml
# Cargo.toml
zeroize = { version = "1", features = ["derive"] }
```
```rust
use zeroize::{Zeroize, ZeroizeOnDrop};

#[derive(ZeroizeOnDrop)]
pub struct KeyManager {
    #[zeroize]
    master_key: [u8; 32],
}
```
Appliquer egalement a `StreamingDecryptor.file_key`.

### S-02 -- MOYEN : Comparaison token COG non constant-time

**Fichier** : `apps/miyucloud/src/api/auth.rs` ligne 80

**Description** : `token == expected` utilise la comparaison standard `String::eq` qui court-circuite.

**Recommandation** : Remplacer par la meme logique XOR accumulateur deja utilisee dans `verify_password()`, ou utiliser `subtle::ConstantTimeEq`.

### S-03 -- MOYEN : Absence de headers de securite HTTP sur la surface web

**Fichier** : `apps/miyucloud/src/web_surface/mod.rs`

**Description** : Aucun header de securite n'est present (CSP, X-Frame-Options, HSTS, X-Content-Type-Options, Referrer-Policy).

**Recommandation** : Ajouter un middleware Tower dans `web_surface_router()` :
```rust
.layer(axum::middleware::map_response(|mut response: Response| async move {
    let headers = response.headers_mut();
    headers.insert("X-Content-Type-Options", "nosniff".parse().unwrap());
    headers.insert("X-Frame-Options", "DENY".parse().unwrap());
    headers.insert("Referrer-Policy", "no-referrer".parse().unwrap());
    headers.insert("Content-Security-Policy",
        "default-src 'self'; script-src 'unsafe-inline'; style-src 'unsafe-inline'".parse().unwrap());
    response
}))
```

### S-04 -- FAIBLE : Header injection dans Content-Disposition

**Fichier** : `apps/miyucloud/src/web_surface/share_page.rs` ligne 257

**Description** : Le nom de fichier est insere sans echappement des guillemets dans le header `Content-Disposition`. Un fichier nomme `test"injected` pourrait casser le header.

**Recommandation** : Echapper les guillemets ou utiliser l'encodage RFC 5987 :
```rust
let safe_name = file_entry.name.replace('"', "\\\"");
let content_disposition = format!("attachment; filename=\"{safe_name}\"");
```
